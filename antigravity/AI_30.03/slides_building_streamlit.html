Part 1

Streamlit Basics

Your first web app in 5 lines of Python

Let us start with the fundamentals. In this section, you will learn the core Streamlit widgets and patterns. By the end, you will be able to create interactive web interfaces using nothing but Python.

Hello World

import streamlit as st

st.title("Hello World!")
name = st.text_input("Your name?")
if name:
    st.write(f"Hi {name}!")

Run with: streamlit run app.py

That's a COMPLETE web app.

No server setup.
No routes.
No templates.
No HTML.

Just. Python.

Here it is. Four lines of real code and you have a web app with a title, a text input, and dynamic output. When you run streamlit run app dot py, it opens a browser, shows the title and the input. You type your name and it greets you. The key insight: Streamlit re-runs your script from top to bottom every time something changes. That is the whole execution model. Simple but powerful.

Every widget returns its current value. Use it like a variable.

age = st.slider("Your age", 0, 100, 25)   # returns integer
st.write(f"You are {age} years old")
Here are the most important widgets. Each one is a single function call that both renders the widget and returns its value. The slider returns a number. The text input returns a string. The checkbox returns true or false. You just assign the result to a variable and use it in your logic. No event handlers, no callbacks, no state management. The variable always holds the current value.

Part 2

Embeddings Page

Loading models, computing similarity, interactive heatmaps

Now we build the first real page of our app. The embeddings explorer lets users type sentences and see how similar they are to each other, visualized as an interactive heatmap.

Caching with @st.cache_resource

@st.cache_resource
def load_model():
    from langchain_huggingface import (
        HuggingFaceEmbeddings
    )
    return HuggingFaceEmbeddings(
        model_name="all-MiniLM-L6-v2"
    )

# First call: loads model (~3 sec)
model = load_model()

# Every call after: instant!
model = load_model()  # cached

Why caching matters

The model is ~80MB.

Without caching: reloads every single interaction.

With caching: loads once, reuses forever.

@st.cache_resource — for models, DB connections
@st.cache_data — for DataFrames, API results

This is possibly the most important Streamlit concept. Remember, Streamlit reruns your entire script on every interaction. If you load an 80MB model at the top of your script, it would reload every time someone moves a slider. The cache resource decorator tells Streamlit to load it once and reuse the same object. There are two caching decorators: cache_resource for heavy objects like models and database connections, and cache_data for serializable data like DataFrames and API results. Always cache your model loading.

Embeddings Page Flow

User types texts
st.text_area()
→
Click button
st.button()

Compute embeddings

Show results
st.plotly_chart()
texts = st.text_area("Enter texts (one per line)",
    "The cat sat on the mat\nThe dog played in the park\nFelines rest on rugs")

if st.button("Compute Embeddings"):
    lines = [t.strip() for t in texts.split("\n") if t.strip()]
    embeddings = model.embed_documents(lines)

    # Compute cosine similarity matrix
    from sklearn.metrics.pairwise import cosine_similarity
    sim_matrix = cosine_similarity(embeddings)

    st.success(f"Computed {len(lines)} embeddings ({len(embeddings[0])}D each)")
Here is the core logic for the embeddings page. The user types sentences into a text area, one per line. When they click the button, we split the text into lines, compute embeddings for each one using the cached model, and then calculate the cosine similarity matrix. The key pattern here is the button guard: the code inside the if block only runs when the user clicks. This prevents expensive computations from running on every keystroke.

Interactive Exploration

# Let users explore with their own texts
col1, col2 = st.columns(2)

Three strategies, interactive comparison, visual results

The question: where do you cut?

Cut badly → lose meaning. Cut well → better search results.

Embedding models have a maximum input length, typically 256 or 512 tokens. A research paper or book chapter is much longer than that. So we need to split documents into chunks. But here is the key insight: where you cut matters enormously. If you cut in the middle of a sentence or an idea, the embedding of that chunk will be noisy and inaccurate. Good chunking preserves semantic boundaries. This directly affects search quality downstream.

Three Strategies

Fixed-Size

Split every N characters

+ Simple and fast
+ Predictable chunk sizes
− Cuts mid-word/sentence
− Loses context

Sentence-Based

Split on sentence boundaries

+ Respects meaning
+ Complete thoughts
− Variable sizes
− Needs sentence detection

LangChain Recursive

Smart separator hierarchy

+ Best of both worlds
+ Respects paragraphs first
+ Falls back gracefully
Separators: ¶ → . → space

In practice: Recursive is the default choice. Use sentence-based for Q&A. Fixed-size only for structured data.

Three approaches, each with tradeoffs. Fixed-size is the simplest: every 200 characters, cut. But it may cut in the middle of the word "artificial" giving you "artifi" in one chunk and "cial" in the next. Sentence-based respects meaning boundaries but gives you variable-sized chunks. LangChain's recursive character splitter is the smartest: it tries to split on paragraph breaks first, then sentence breaks, then spaces, then characters. It gives you the most semantically meaningful chunks. In practice, recursive is what you should default to.

Part 4

Vector Search Page

ChromaDB, semantic search, the full pipeline

Now we put it all together. In this section, we build a vector search page that chunks documents, embeds them, stores them in ChromaDB, and lets users search semantically. This is the core of every RAG system.

ChromaDB in 3 Lines

from langchain_community.vectorstores import Chroma

# Create a searchable vector database
vectorstore = Chroma.from_texts(
    texts=chunks,                    # your chunked documents
    embedding=load_model(),          # cached embedding model
    collection_name="my_docs"
)

# Search!
results = vectorstore.similarity_search_with_relevance_scores(
    query="How does photosynthesis work?",
    k=3                              # return top 3 results
)

1 Give it texts and an embedding model 2 It embeds and stores automatically 3 Query with natural language

ChromaDB through LangChain is remarkably simple. Three lines: create from texts with an embedding model, then search. Under the hood, Chroma embeds all your chunks, stores them as vectors, and when you query, it embeds your query and finds the nearest neighbors. The similarity_search_with_relevance_scores variant also returns how relevant each result is, which we will use to show confidence bars in the UI.

Search Interface

query = st.text_input("Search your documents", "What is machine learning?")
k = st.slider("Number of results", 1, 10, 3)

if st.button("Search") and query:
    results = vectorstore.similarity_search_with_relevance_scores(query, k=k)

    for i, (doc, score) in enumerate(results):
        with st.expander(f"Result {i+1} (relevance: {score:.2%})", expanded=(i==0)):
            st.markdown(doc.page_content)
            st.progress(float(score))          # visual relevance bar

            if score > 0.8:
                st.success("High relevance")
            elif score > 0.5:
                st.info("Moderate relevance")
            else:
                st.warning("Low relevance")

Each result is an expander with the text, a relevance bar, and a colored indicator.

The search interface is clean and informative. A text input for the query, a slider for how many results to show, and a search button. Each result appears in an expander. The first result is expanded by default. Inside, we show the actual text, a progress bar as a visual relevance indicator, and a colored status message. This gives users an immediate sense of how confident the search is. The pattern of using expanders for results keeps the page clean even with many results.

This is the foundation of every RAG (Retrieval-Augmented Generation) system.

ChatGPT + your documents = RAG. The retrieval part is exactly what we built.

# The entire pipeline in Streamlit
text = st.text_area("Paste your document")
chunks = splitter.split_text(text)                            # Chunk
vectorstore = Chroma.from_texts(chunks, embedding=model)      # Embed + Store
results = vectorstore.similarity_search(query, k=3)           # Query + Retrieve
Step back and see the full picture. Document goes in, gets chunked, each chunk gets embedded into a vector, vectors get stored in ChromaDB, user asks a question, the question gets embedded, nearest vectors are found, and the matching chunks are returned. This is exactly how retrieval-augmented generation works. When ChatGPT searches your documents, it does this pipeline. You have just built the retrieval half. Add an LLM on top to generate answers from the retrieved chunks, and you have a complete RAG system.

Streamlit Cloud

2 Go to "Secrets"

Same st.secrets API works in both environments.

Never hardcode API keys in your code. Streamlit has a built-in secrets management system. Locally, you create a secrets.toml file in the .streamlit directory. In production on Streamlit Cloud, you paste the same TOML content into the app settings. Your code uses st.secrets with the key name, and it works identically in both environments. Always add your secrets file to gitignore so it never gets committed to version control.
