Types of Machine Learning

Supervised Learning

Learning with labeled data.

A teacher shows the correct answers. The model learns to map inputs to known outputs.

Example: "This image is a cat."
Model learns: image -> cat.

Unsupervised Learning

Finding patterns in unlabeled data.

No teacher. The model discovers structure on its own through self-discovery.

Example: grouping customers
by purchase behavior.

Reinforcement Learning

Learning through trial and error.

An agent takes actions, receives rewards or penalties, and learns a strategy.

Example: a robot learning
to walk by falling and adjusting.

Today we focus on supervised learning (Hour 1) and unsupervised patterns via embeddings (Hour 2).

These are the three big families of machine learning. Supervised learning is the most common: you have data with labels and you train a model to predict those labels. Unsupervised learning finds hidden structure without labels, like clustering. Reinforcement learning is about an agent learning from trial and error, like training a robot or playing a game. Today we will mainly work with supervised learning when we train a DNN, and we will use embeddings which are learned through a form of unsupervised or self-supervised learning.

PyTorch in 60 Seconds

Why PyTorch?

Industry standard for research & production
Dynamic computation graph
Pythonic and intuitive
Powers: Meta AI, Tesla, OpenAI

Core idea: Tensors + Autograd 

import torch
import torch.nn as nn

# Tensors = NumPy arrays with GPU support
x = torch.randn(32, 784)  # batch of 32

# Autograd = automatic differentiation
w = torch.randn(784, 10, requires_grad=True)
y = x @ w
loss = y.sum()
loss.backward()  # gradients computed!
print(w.grad.shape)  # (784, 10)
PyTorch is the most popular deep learning framework. Think of tensors as NumPy arrays that can also run on GPUs and automatically track gradients. Autograd means you define the forward computation and PyTorch figures out how to compute the gradients for you. This is what makes training neural networks practical. You don't have to derive the math by hand.

What is a Neuron?

A neuron takes inputs, multiplies by weights, adds them up, and applies an activation function.

x1
input
x2
input
x3
input
→ w1 → w2 → w3
Σ + b
weighted sum
→
f(x)

→
y
output

y = f(w1*x1 + w2*x2 + w3*x3 + b) — that's it. Stack many neurons → neural network.

This is the fundamental building block. A single neuron takes inputs, multiplies each by a weight, sums them up, adds a bias, and passes the result through an activation function. The activation function introduces non-linearity, which is what lets neural networks learn complex patterns. Stack many neurons in layers, and you get a neural network. The weights and biases are the parameters that the network learns during training.

Building a Neural Network

class FashionNet(nn.Module):
    def __init__(self):
        super().__init__()
        self.network = nn.Sequential(
            nn.Flatten(),                  # 28x28 image -> 784 numbers
            nn.Linear(784, 256),           # Hidden layer 1 (256 neurons)
            nn.ReLU(),                     # Activation function
            nn.Dropout(0.3),               # Regularization (prevent memorizing)
            nn.Linear(256, 128),           # Hidden layer 2 (128 neurons)
            nn.ReLU(),
            nn.Linear(128, 10)             # Output: 10 clothing categories
        )

    def forward(self, x):
        return self.network(x)

model = FashionNet()
print(f"Parameters: {sum(p.numel() for p in model.parameters()):,}")
# Parameters: 235,146

nn.Sequential chains layers together / 235K learnable parameters / 10 output classes for Fashion-MNIST

Here is our neural network. It takes a 28 by 28 grayscale image, flattens it into 784 numbers, passes through two hidden layers with ReLU activation, and outputs 10 numbers representing 10 clothing categories. Dropout randomly disables 30 percent of neurons during training, which prevents the model from memorizing the training data. The model has about 235 thousand parameters. This is tiny by modern standards, but plenty for learning clothing recognition.

The Training Loop

Every DNN training follows the same 4 steps, repeated many times:

1. Forward Pass
Push data through network
→
2. Compute Loss
How wrong are we?
→
3. Backward Pass
Compute gradients
→
4. Update Weights
Adjust to reduce error 
optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
criterion = nn.CrossEntropyLoss()

for epoch in range(10):
    for images, labels in train_loader:
        outputs = model(images)           # 1. Forward
        loss = criterion(outputs, labels)  # 2. Loss
        optimizer.zero_grad()             # Clear old gradients
        loss.backward()                   # 3. Backward (autograd!)
        optimizer.step()                  # 4. Update weights

This exact loop is used for everything from a 3-layer MLP to GPT-4. Only scale changes.

This is the heart of deep learning. Step one: push your data through the network to get predictions. Step two: compare predictions to the real labels using a loss function, which gives you a single number measuring how wrong the model is. Step three: backpropagation, where PyTorch automatically computes how each weight contributed to the error. Step four: the optimizer adjusts all weights slightly to reduce the error. Repeat thousands of times. This same loop is used to train every deep learning model you have heard of.

Key Decisions When Training

Activation

ReLU — default for hidden layers
Sigmoid — binary output
Softmax — multi-class probabilities

Rule: ReLU for hidden,
match output to task.

Loss Function

CrossEntropyLoss — classification
MSELoss — regression
BCELoss — binary tasks

Must match your task.
Wrong loss = wrong learning.

Optimizer

Adam — adaptive, great default
SGD — simpler, needs tuning
AdamW — Adam + weight decay

Start with Adam(lr=1e-3).
Tune only if needed.

Three decisions you make for every training run. Activation functions: ReLU is almost always correct for hidden layers. Loss function: must match your problem type. CrossEntropyLoss for multi-class classification, which is what we use with Fashion-MNIST. Optimizer: Adam is the safe default. It adapts the learning rate automatically. These are the practical choices that turn theory into working code.

Evaluating Your Model

# Switch to evaluation mode (disables Dropout)
model.eval()
correct, total = 0, 0

with torch.no_grad():           # No gradient tracking needed
    for images, labels in test_loader:
        outputs = model(images)
        _, predicted = outputs.max(1)  # Get predicted class
        total += labels.size(0)
        correct += (predicted == labels).sum().item()

accuracy = correct / total
print(f"Test Accuracy: {accuracy:.1%}")  # e.g., 89.3%

model.train()  # Switch back for further training

model.eval() disables Dropout and BatchNorm / torch.no_grad() saves memory / Always test on unseen data!

After training, you need to evaluate on data the model has never seen. Two critical details: model.eval puts the network in evaluation mode, which disables dropout and changes batch normalization behavior. torch.no_grad tells PyTorch not to track gradients, saving memory. A common mistake is forgetting model.eval, which gives you incorrect accuracy numbers.

What Are Embeddings?

GPS Coordinates for Meaning

Just as GPS coordinates represent a location in physical space...

Embeddings represent the meaning of text in mathematical space.

Similar meanings → nearby coordinates.
Different meanings → far apart.

Text → Vector (numbers)

"A happy dog"
→ [0.23, -0.41, 0.87, ...]

"A joyful puppy"
→ [0.25, -0.39, 0.84, ...] ✓ close!

"Quantum physics"
→ [-0.71, 0.52, -0.13, ...] ✗ far
Embeddings are one of the most powerful ideas in modern AI. Think of them as GPS coordinates, but for meaning instead of location. A pre-trained model has learned to convert text into a list of numbers, a vector, such that similar texts get similar numbers. "A happy dog" and "a joyful puppy" have almost the same embedding, even though the words are completely different. "Quantum physics" has a very different embedding because the meaning is unrelated. This enables searching by meaning rather than by keywords.

Creating Embeddings with LangChain

LangChain gives us a clean API for working with embedding models. Here we use all-MiniLM-L6-v2, a small but effective model that converts any text into a vector of 384 numbers. It was pre-trained on over a billion sentence pairs, so it has learned rich representations of meaning. You can embed a single query or batch-embed multiple documents. The key point: someone else did the heavy training. We just use the result.

Measuring Similarity

Cosine Similarity

Measures the angle between two vectors.

1.0 = identical meaning
0.0 = unrelated
-1.0 = opposite meaning

Like comparing the direction two arrows point, ignoring length. 

from sklearn.metrics.pairwise \
    import cosine_similarity
import numpy as np

texts = [
    "I love machine learning",
    "AI and ML are fascinating",
    "The weather is sunny today"
]
vecs = embeddings.embed_documents(texts)

# Similarity matrix
sim = cosine_similarity(vecs)
# [1.00, 0.82, 0.13]  <- ML ~ AI
# [0.82, 1.00, 0.11]  <- AI ~ ML
# [0.13, 0.11, 1.00]  <- weather ≠ ML
Once you have embeddings, you can compare them using cosine similarity. It measures the angle between two vectors: 1 means identical direction, 0 means unrelated, minus 1 means opposite. In our example, "I love machine learning" and "AI and ML are fascinating" get a high score of 0.82 even though they share few words. "The weather is sunny today" scores very low against both. This is the foundation of semantic search: find documents whose embeddings are closest to the query's embedding.

Why Chunking Matters

Real documents are too long for embedding models. We need to split them into chunks.

Long Document (5,000 words)
Too large for embedding model
(max ~512 tokens)
→
Chunk 1 (200 words)
Chunk 2 (200 words)
Chunk 3 (200 words)
... more chunks
Smaller chunks
= more precise retrieval
= better search results
Too small
= lose context
= fragments of ideas
Sweet spot
= 200-500 words per chunk
= complete thoughts
Embedding models have a maximum input length, typically around 512 tokens. But real documents, like articles, reports, or textbooks, are much longer. We need to split them into smaller chunks before embedding. This is called chunking. The chunk size matters: smaller chunks give more precise retrieval because each chunk covers a focused topic. But if chunks are too small, they lose context and become fragments. The sweet spot is usually 200 to 500 words per chunk, which captures complete thoughts or paragraphs.

Chunking Strategies

Fixed-Size

Split every N characters.

Pros: Simple, fast, predictable chunk sizes.

Cons: May break mid-sentence or mid-word.

text[0:500], text[500:1000]...

Sentence-Based

Split at sentence boundaries.

Pros: Respects meaning, clean breaks.

Cons: Variable chunk sizes, may be too small.

nltk.sent_tokenize(text)

Recursive (LangChain)

Try splitting by paragraphs, then sentences, then words.

Pros: Smart, flexible, best quality.

Cons: Slightly more complex.

RecursiveCharacterTextSplitter

Three main strategies. Fixed-size is the simplest: just chop the text every 500 characters. It is fast but can break a sentence right in the middle, which hurts the quality of embeddings. Sentence-based splitting respects natural boundaries but gives you variable-sized chunks. LangChain's recursive splitter is the smartest: it first tries to split by paragraphs. If a paragraph is still too long, it splits by sentences. If a sentence is too long, it splits by words. This gives you the best quality chunks because it preserves meaning at every level.

Chunking with LangChain

from langchain.text_splitter import RecursiveCharacterTextSplitter

# The recommended approach: recursive splitting
splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # Max characters per chunk
    chunk_overlap=50,     # Overlap between chunks (preserve context)
    separators=["\n\n", "\n", ". ", " ", ""]  # Try these in order
)

# Split a long document
document = "..."  # Your long text
chunks = splitter.split_text(document)

print(f"Document: {len(document)} chars -> {len(chunks)} chunks")
# Document: 5000 chars -> 12 chunks

# Each chunk preserves complete thoughts
for i, chunk in enumerate(chunks):
    print(f"Chunk {i}: {len(chunk)} chars")

chunk_overlap: chunks share some text at boundaries → prevents losing context between chunks.

Here is how to use LangChain's recursive splitter in practice. You specify the maximum chunk size and the overlap. The overlap is important: it means consecutive chunks share some text at their boundaries, so you don't lose context. The separators list tells LangChain to first try splitting by double newlines (paragraphs), then single newlines, then periods (sentences), then spaces (words), and only as a last resort by character. This gives you high-quality chunks that respect the document's natural structure.

Choosing a Strategy

Strategy Best For Chunk Quality Speed
Fixed-Size Quick prototyping, uniform data Low (breaks sentences) Fastest
Sentence-Based Short documents, Q&A data Medium (clean boundaries) Fast
Recursive Long documents, articles, reports High (preserves meaning) Fast
Semantic Research, high-quality RAG Highest (topic-aware) Slower (uses embeddings)

Rule of thumb: start with Recursive. Only switch if you have a specific reason.

This table summarizes when to use each strategy. For most use cases, recursive splitting is the best default. It is fast, produces high-quality chunks, and handles a wide variety of document types. Semantic chunking is the most advanced: it uses embeddings to detect topic boundaries within the text. It gives the highest quality but is slower because it needs to compute embeddings for every potential split point. My recommendation: start with recursive and only move to semantic chunking if your search quality needs improvement.

ChromaDB: Vector Database

What is a Vector DB?

A database optimized for storing and searching embedding vectors.

Traditional DB: WHERE text LIKE '%keyword%'

Vector DB: "Find documents similar in meaning to this query."

ChromaDB: open-source, Python-native, simple API. 

from langchain_community.vectorstores \
    import Chroma

# Store documents (auto-embeds!)
vectorstore = Chroma.from_texts(
    texts=chunks,
    embedding=embeddings
)

# Semantic search
results = vectorstore.similarity_search(
    "How do computers learn?",
    k=3  # top 3 results
)

for doc in results:
    print(doc.page_content[:100])
ChromaDB is a vector database designed for AI applications. Unlike a traditional database where you search by keywords, a vector database searches by meaning. You store your document chunks along with their embeddings, and then you can query with natural language. ChromaDB finds the chunks whose embeddings are closest to the query's embedding. Through LangChain, embedding is automatic. You just pass your texts and the embedding model, and ChromaDB handles the rest.

This is exactly how RAG systems work, the technology behind tools like ChatGPT's document understanding, Copilot's codebase search, and many other AI applications. And it is just six lines of code.

Hour 3

🔬 DNN vs Embeddings

Comparison / Decision Guide / Transfer Learning

Welcome to the final hour. Now that you have built both a DNN from scratch and an embedding pipeline, the natural question is: when should you use which? Hour 3 answers that question with a direct comparison, a decision guide, and a look at the transfer learning spectrum.

DNN vs Embeddings: Side by Side

DNN from Scratch Embeddings + ML
Training time Minutes to hours (GPU) Seconds to minutes (CPU)
Data needed 10,000+ labeled examples 100-5,000 examples
Flexibility Full control over architecture Limited to pre-trained features
Performance Can be highest with enough data Often surprisingly competitive
Expertise needed Deep learning knowledge Basic ML + API skills
Best for Novel tasks, massive datasets Text tasks, rapid prototyping

Key insight: embeddings + simple ML is often 90% as good in 10% of the time.

Here is the direct comparison. Training a DNN gives you full control but requires more data, more compute, and more expertise. Embeddings plus a simple classifier gives you competitive results much faster, especially for text tasks. The key insight is that embeddings plus simple ML often gets you 90 percent of the performance in 10 percent of the time. Start with embeddings. Only move to a custom DNN if you have a specific reason: novel data types, massive datasets, or the need to squeeze out every last bit of accuracy.

Here is a practical decision framework. Always start with the simplest approach: pre-trained embeddings plus a simple classifier. If that does not give you sufficient accuracy, try fine-tuning a pre-trained model on your specific data. Only train from scratch if you have a truly novel task that no existing model covers, and you have enough data and compute to justify it. In practice, most real-world problems are solved in the first or second box. Training from scratch is much rarer than you might expect.

Embeddings Are Superpowers

GPS coordinates for meaning. Pre-trained models give you features for free. Chunk → embed → search or classify.

Chunking Matters

Right chunks = right answers. Recursive splitting is the best default. Overlap preserves context at boundaries.

Know When to Use What

Start simple (embeddings + ML). Scale up (fine-tune / DNN) only when needed. Speed of iteration wins.

Four things to remember from today. First: the training loop is universal across all of deep learning. Second: embeddings give you rich features from pre-trained models for free. Third: chunking is a critical step that affects the quality of everything downstream. Fourth: start simple and only scale up complexity when you need to. These principles will serve you well in any ML project.
